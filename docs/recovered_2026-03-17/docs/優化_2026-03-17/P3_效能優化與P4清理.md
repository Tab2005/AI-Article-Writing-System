## 變更詳情

### P3-1：AI 配置快取 TTL 機制

**檔案**：`seonize-backend/app/services/ai_service.py`

- 為 `AIService.get_config()` 新增了 60 秒的 TTL (Time-To-Live) 快取機制。
- 透過 `time.monotonic()` 判斷快取效力，若超過 60 秒或 `invalidate_config()` 被呼叫，將主動從資料庫重新載入最新配置。
- **優點**：管理員更改 AI 提供者或 API Key 後，無需重啟服務即可在 1 分鐘內自動生效。

### P3-2：Kalpa 批量編織 Session 安全

**檔案**：`seonize-backend/app/services/kalpa_service.py`

- 徹底解決了協程並發下共用一個 SQLAlchemy Session 的競爭問題。
- 現改為在異步 `Task` 內部透過 `get_db_context()` 建立獨立的資料庫連接 Session。
- **優點**：避免了並行任務間因交易衝突導致的資料寫入失敗或連線崩潰。

### P3-3：前端 API 請求取消支援

**檔案**：`seonize-frontend/src/services/api.ts`

- 在 `apiFetch` 體系中引入了 `signal` (AbortSignal) 參數。
- 優化了超時與外部信號的整合邏輯（使用 `AbortSignal.any`），並能正確區分「手動取消」與「系統超時」的錯誤訊息。
- **優點**：為後續前端元件「切換分頁主動停止請求」提供了底層基礎。

## 操作方式變更

- **後端 API**：無外部接口變更。
- **前端開發**：現在調用 API 服務時，可視需求傳入 `signal` 參數，例如：
  ```typescript
  useEffect(() => {
    const controller = new AbortController();
    api.search.serp(data, { signal: controller.signal });
    return () => controller.abort();
  }, []);
  ```
